# PaperMentor OS 产品需求文档

## 1. 文档目的

本 PRD 用于定义 PaperMentor OS 的稳定 V1 范围。

它主要服务于：

- 产品定义
- 比赛方案准备
- 团队开发对齐
- 后续任务拆解

本文件刻意选择“范围收敛但论证扎实”的 MVP，而不是一个看起来很大但落地性差的概念。

## 2. 产品概述

### 2.1 产品名称

PaperMentor OS

### 2.2 一句话定位

PaperMentor OS 是一个多智能体论文评审与指导系统，用于从多个维度诊断学生论文中的问题，并在默认不代写的前提下输出结构化修改建议。

### 2.3 产品愿景

帮助大学生提升论文研究深度与写作质量，同时减轻导师在前期审查阶段的部分工作负担。

### 2.4 产品使命

把 AI 从“答案生成器”变成“研究指导系统”。

## 3. 问题定义

### 3.1 核心用户问题

很多学生难以准确判断：

- 自己的选题是否真的有意义、范围是否合理
- 自己的论文是否有清晰的研究问题
- 自己的论证链是否完整
- 自己的文献支撑是否充分
- 自己宣称的创新点是否真实且有足够支撑

与此同时，导师往往无法在论文早期为每位学生持续提供细粒度反馈。

### 3.2 为什么现有 AI 工具不够

常见 AI 写作工具通常会：

- 直接给答案，而不是先诊断问题
- 过度改写文本
- 不能系统性覆盖多个评审维度
- 把评审标准藏在单个 prompt 中，难以维护
- 在主观判断维度上表现不稳定

### 3.3 为什么值得现在做

- AI+教育是一个很强的比赛方向
- 论文质量与研究深度不足是普遍且容易说明的痛点
- multi-agent 架构已经足以支撑更有说服力的结构化评审

## 4. 目标用户

### 4.1 主要用户

- 正在撰写课程论文、项目报告或毕业论文的本科生

### 4.2 次要用户

- 准备研究型报告的学生团队
- 希望减轻重复性初审工作的导师
- 希望得到结构化初审工具的任课教师

### 4.3 V1 初始用户群

V1 初始用户建议聚焦于：

- 计算机专业本科毕业论文作者

原因：

- 更容易定义领域适配规则
- 更容易收集样本文本
- 更容易构建评测用例

## 5. 使用场景

### 5.1 场景 A：论文草稿初审

学生上传论文草稿后，系统输出：

- 多维度诊断结果
- 高优先级问题清单
- 具体的下一步修改建议

### 5.2 场景 B：开题报告或任务书检查

学生上传开题材料后，系统输出：

- 选题与范围评估
- 文献缺口提示
- 可行性与创新风险提示

### 5.3 场景 C：导师初筛辅助

导师查看报告后，可以快速看到：

- 主要问题所在
- 薄弱维度集中在哪些部分
- 哪些点值得人工进一步重点跟进

## 6. 产品目标

### 6.1 用户目标

- 知道当前论文到底问题出在哪里
- 知道这些问题为什么重要
- 知道下一步应该优先改什么
- 保留自己对论文写作与研究过程的主导权

### 6.2 产品目标

- 在固定维度上提供系统化评审
- 生成可执行、可排序的指导建议
- 保持结果可解释、可追溯
- 避免产品滑向一键代写工具

### 6.3 比赛目标

- 清晰体现教育价值
- 清晰论证为什么需要 multi-agent
- 清晰展示 skill-based 架构的可维护性

## 7. 成功指标

### 7.1 一级指标

- 人工评审认为系统能够识别草稿中的主要问题，尤其是高严重度问题
- 人工评审认为输出建议具有可操作性
- 人工评审认为系统具备多维度评审覆盖能力
- 人工评审认为系统结论具备基本专业合理性
- 用户反馈系统是在帮助自己思考，而不只是生成答案

### 7.2 二级指标

- 多次运行结果的一致性
- 报告中无依据结论的比例较低
- 违反“不代写”策略的比例较低

### 7.3 比赛演示指标

- 能展示明确的“修改前 / 修改后”对比
- 能展示与单模型 baseline 的差异
- 能用易懂方式解释系统架构
- 能以计算机专业本科毕业论文草稿作为固定演示样例

## 8. 产品原则

### 8.1 指导优先

产品应该先做诊断和指导，而不是先替用户生成文本。

### 8.2 证据优先

每一个重要结论都应尽量对应论文中的具体位置或证据缺失模式。

### 8.3 范围收敛但深度足够

V1 应该把少量关键维度做好，而不是把很多维度都做浅。

### 8.4 角色稳定，Skill 可演进

核心 agent 角色应保持稳定，而 rubric 与 policy 通过 skills 进行演进。

## 9. 范围定义

### 9.1 V1 范围内

- 计算机专业本科毕业论文文本输入
- 章节与参考文献的结构化解析
- 固定五维评审
- 在主观维度上进行选择性 debate
- 输出结构化评审报告
- 基于 skill 的 rubric 与 policy 加载
- 基于证据的 findings
- `docx` 输入支持
- 按维度输出 `10` 分制评审结果
- 输出可导出的 `PDF` 报告文件
- 高优先级问题按“研究内容相关 > 文本表达相关”排序

### 9.2 V1 范围外

- 自动完整重写论文
- 比赛演示时依赖在线检索最新文献
- 对所有技术细节做深层事实核验
- 一次性支持所有学科
- 导师与学生多角色工作流管理
- 评论协作、版本历史等平台型功能
- `pdf` 输入解析
- 开题报告作为 V1 主场景

## 10. 功能需求

### 10.1 输入管理

系统必须支持：

- 标题输入
- 摘要输入
- 正文输入
- 按章节组织的论文内容
- 参考文献输入
- 论文阶段选择
- 可选的学科选择
- `docx` 文件输入

系统建议支持：

- 学校格式要求选择
- 导师个性化要求说明
- 后续扩展 `pdf` 文本解析

### 10.2 评审引擎

系统必须覆盖以下五个评审维度：

1. 选题价值与问题清晰度
2. 逻辑与论证链路
3. 文献支撑
4. 创新性与研究深度
5. 写作与格式规范

### 10.3 多智能体编排

系统必须能够：

- 将评审任务分发给专项 agent
- 为各 agent 加载匹配的 rubric 与 policy skills
- 合并结构化输出
- 检测主观模块中的结论分歧
- 在达到阈值时触发 debate

### 10.4 Evidence Ledger

系统必须能够：

- 用统一 schema 存储所有 findings
- 为 findings 绑定 evidence anchor
- 记录来源 worker 与 skill 版本
- 为最终报告生成提供统一数据入口

### 10.5 报告输出

最终报告必须包含：

- 总体总结
- 各维度结果
- 高优先级问题
- 问题重要性的解释
- 修改顺序建议
- 面向学生的下一步指导
- 面向导师的快速查看摘要
- 各维度 `10` 分制评分
- 可导出 `PDF` 文件

### 10.6 安全与边界控制

系统必须：

- 区分 review mode 与 writing mode
- 默认避免整段式代写
- 当用户尝试绕过 review mode 时给出明确提示

## 11. 非功能需求

### 11.1 可解释性

系统应尽可能让每个重要结论都可以追溯来源。

### 11.2 可维护性

rubric 与 policy 应该可以通过 skills 进行更新，而不改动整体 agent 架构。

### 11.3 一致性

系统应尽量降低主观评审维度上的不稳定输出。

### 11.4 演示稳定性

系统应在比赛可接受的时间窗口内稳定输出完整报告。

## 12. 用户流程

### 12.1 端到端流程

1. 用户选择论文阶段和学科
2. 用户上传或粘贴论文内容
3. 系统将输入解析为 `PaperPackage`
4. `ChiefReviewer` 选择流程和 skills
5. 专项 agents 并行评审
6. 主观模块进行分歧检测
7. 有需要时触发 debate
8. `GuidanceComposer` 生成最终报告
9. 用户阅读带优先级的修改指导

## 13. 输出规范

### 13.1 报告结构

- 整体健康度概览
- 各维度评分
- 最紧急的三个问题
- 各维度 findings
- 基于证据的解释
- 修改优先级排序
- 下一步建议

补充规则：

- 报告按维度展开
- 评分采用 `10` 分制
- 高优先级问题先展示研究内容相关问题，再展示文本表达相关问题
- 最终报告需支持导出为 `PDF`

### 13.2 语气要求

报告应该呈现为：

- 建设性
- 严谨
- 具有教学引导感

报告不应该呈现为：

- 普通聊天机器人回答
- 已经替学生改好的论文
- 空泛鼓励式文本

## 14. Agent 需求

### 14.1 V1 固定 Agent 集合

- `ChiefReviewer`
- `TopicScopeAgent`
- `LogicChainAgent`
- `LiteratureSupportAgent`
- `NoveltyDepthAgent`
- `WritingFormatAgent`
- `DebateJudgeAgent`
- `GuidanceComposer`

### 14.2 Agent 责任要求

每个 agent 必须具备：

- 稳定的职责边界
- 明确的输出 schema
- 基于 skill 的行为约束
- 不允许脱离编排流程私自交叉调用

## 15. Skill 需求

V1 必须支持：

- rubric skills
- policy skills
- output schema skills
- 至少一个面向计算机专业的 domain adaptation skill

每个 skill 必须具备：

- metadata
- version
- prompt 或规则定义
- output schema 绑定
- evaluation cases

## 16. 风险

### 16.1 产品风险

- 用户仍可能期待系统直接帮自己写论文
- 用户可能把系统建议误解成绝对权威
- 范围过宽会削弱演示质量

### 16.2 技术风险

- 主观判断仍可能存在不稳定性
- 输入解析质量会影响下游评审质量
- 多智能体编排可能带来延迟

### 16.3 比赛风险

- 如果评测薄弱，架构容易被认为是过度设计
- 如果输出太长，现场演示效果会下降
- 如果不能明确优于单 prompt baseline，差异化会变弱

## 17. 风险控制

- 严格控制 V1 范围
- 强制 evidence anchor
- 只在真正有价值的模块上启用 debate
- 尽早建立离线评测样本
- 必须与单模型 baseline 做对比

## 18. MVP 定义

当系统能够做到以下几点时，可认为 V1 MVP 完成：

- 接收一篇计算机专业本科毕业论文 `docx` 草稿
- 跑完整的五维评审
- 生成结构化报告
- 能对主要问题给出基于证据的解释
- 默认不滑向代写
- 能输出按维度组织、10 分制展示、可导出 PDF 的报告

超出这些能力的部分，属于增强项，而不是 MVP 本体。

## 19. 里程碑

### 里程碑 1：规格冻结

- 完成 PRD 定稿
- 完成架构定稿
- 完成报告 schema 定稿

### 里程碑 2：核心链路

- 输入解析
- agent 编排
- evidence ledger
- 报告输出

### 里程碑 3：Skill 系统

- rubric skills
- policy skills
- schema skills

### 里程碑 4：评测

- baseline 对比
- 重复运行稳定性测试
- 演示样例准备

### 里程碑 5：比赛包装

- 演示 UI
- 典型样例
- 架构图
- 价值主张表达

## 20. 最终产品表述

PaperMentor OS V1 不是一个泛化的学术万能 AI。
它是一个受控的论文评审与研究指导系统，目标是通过结构化 multi-agent 评估提升学生的研究思考质量。
